# yh-mp-preview v3 升级方案

> 文档状态：已确认范围
> 规划基线：v2.0.10
> 目标版本：v3.0.0
> 目标许可证：AGPL-3.0-or-later
> 计划周期：单人全职约 6–8 周

## 1. 决策摘要

v3 将把 yh-mp-preview 从“Markdown 预览加主题样式”升级为“面向微信公众号的本地确定性排版工作台”。

本次升级确认以下边界：

- 不接入任何 AI 或大模型服务。
- 不实现 AI 排版、AI 写作、AI 主题生成，以及根据参考图片生成主题。
- 不新增生成式图片能力。
- 保留现有本地长图导出，并增加确定性的分段图片导出；这两项属于截图导出，不属于生成式图片能力。
- 不在插件中内嵌第二套富文本编辑器，Obsidian 当前笔记仍是唯一内容源。
- v3 从第一个预发布版本开始采用 AGPL-3.0-or-later。
- 以本地规则、主题组件、文章配方和严格验证保证输出稳定，不依赖网络服务。

v3 使用主版本号升级，是因为主题协议、渲染管线、设置数据结构、复制门禁和许可证都会发生实质变化，不适合继续作为 2.x 的小版本发布。

## 2. 现状基线

当前 v2.0.10 的有效基础：

- TypeScript 全量检查通过。
- 内置 94 个主题 JSON，其中顶层 64 个均能被 Node 正常解析。
- 已支持实时预览、主题切换、背景、字体、页头页尾、中文排版优化、微信公众号富文本复制和长图导出。
- 已支持目录、步骤、清单、摘要卡、作者卡、关注引导、FAQ、时间线、对比表格、对话和图库等本地组件。
- 插件完全本地运行，不会改写原始 Markdown。

当前必须解决的问题：

- 没有自动化测试和固定测试文章。
- 没有专门面向微信公众号的 HTML/CSS 验证器。
- 复制阶段主要删除 `class`、`id` 和 `data-*`，没有系统地处理不兼容标签及 CSS。
- 部分增强组件使用 `div`、`flex`、横向滚动、硬编码强调色等脆弱结构。
- 主题是“元素样式集合”，缺少统一设计令牌、语义组件、文章配方和版本迁移协议。
- `CreateTemplateModal.ts`、`view.ts`、设置面板等文件体积过大，继续叠加功能会增加维护成本。
- 终端输出曾出现编码误判；文件内容经 UTF-8/Unicode 复核正常，后续只增加文本与 JSON 解析防护，不进行全仓转码。
- `package.json` 声明了 `version-bump.mjs`，但仓库中不存在该文件，版本脚本不可复现。
- 发布工作流只上传 `main.js`、`manifest.json`、`styles.css`，缺少许可证、版权声明和构建来源信息。
- 30 个 `xiaohu` 主题由本地临时目录转换而来，需要补齐上游仓库、许可证和修改记录。
- 仓库历史包含多名贡献者，AGPL 迁移必须保留原 MIT 许可和版权声明。

## 3. 产品目标与非目标

### 3.1 产品目标

1. 对同一篇 Markdown、同一主题和同一设置，稳定生成相同 HTML。
2. 复制前明确告诉用户输出是否适合粘贴到微信公众号。
3. 主题不仅定义标题和段落样式，还能定义完整组件和文章结构配方。
4. 保证标题、段落、列表、链接和图片不会在转换过程中静默丢失。
5. 旧版设置和自定义主题可迁移，并提供安全回退。
6. 所有核心能力离线可用，不发送文章和图片到网络。
7. 以 AGPL 方式发布可复现的源代码、构建产物和许可证信息。

### 3.2 非目标

- AI 写作、续写、摘要和润色。
- AI 排版决策或让模型直接生成 HTML。
- AI 主题、图片或插图生成。
- 根据参考图片自动还原主题。
- 云端账号、云同步、在线模板市场。
- 替代微信公众号后台的发布、草稿或素材管理功能。
- 在 Obsidian 内重复实现 Markdown 或富文本编辑器。

### 3.3 计划自检结论

本方案已按以下方面完成一致性检查：

- 范围一致：架构、阶段任务和验收标准均未包含 AI、模型调用或生成式图片；长图和分段图被明确限定为本地截图导出。
- 版本一致：v2.0.11 只做稳定性收口并作为最后一个 MIT 版本；所有 v3 预发布和正式版本均采用 AGPL。
- 许可证顺序一致：只有在 v3 分支完成 AGPL、NOTICE 和来源登记后，才允许复用兼容的 AGPL 代码或主题。
- 兼容策略完整：设置、主题和渲染均有 v2 适配器、迁移备份和回退路径。
- 工期一致：各阶段合计 29–38 个工作日，对应单人全职约 6–8 周。
- 验收可执行：每个阶段都有交付物和退出标准，正式发布另有统一门禁。
- 未决风险未被当成前提：30 个 `xiaohu` 主题的来源仍需核验；在核验完成前，方案不假定它们能够进入 v3 正式发行包。

## 4. 目标工作流

```text
Obsidian Markdown
       ↓
文章模型 ArticleModel
       ↓
本地规则生成 LayoutPlan
       ↓
主题、组件和文章配方
       ↓
确定性 WeChatRenderer
       ↓
兼容性清洗与 ValidationReport
       ↓
预览 / 复制 / HTML / 长图 / 分段图 / 历史快照
```

用户默认流程：

1. 在 Obsidian 中正常编辑 Markdown。
2. 打开 yh-mp-preview，选择主题和文章配方。
3. 查看实时预览及兼容性状态。
4. 修复阻断错误或接受非阻断警告。
5. 复制到微信公众号后台，或导出 HTML、长图、分段图。
6. 必要时从历史快照恢复上一次排版设置。

## 5. 技术架构

建议按职责拆分为以下模块：

```text
src/
  core/
    article/          Markdown DOM → ArticleModel
    layout/           本地排版规则与 LayoutPlan
    theme/            ThemeManifest、注册、迁移、导入导出
    components/       语义组件及确定性渲染器
    render/           WeChatRenderer
    validation/       标签、属性、CSS、内容完整性验证
    clipboard/        复制、HTML 导出和图片资源处理
    history/          排版快照、恢复和数据迁移
  ui/
    view/             主工作台
    theme-studio/     手工主题编辑器
    validation/       检查结果及修复操作
    history/          历史列表和恢复界面
  legacy/
    v2-renderer/      v2 渲染兼容层
    v2-theme/         旧主题适配器
```

### 5.1 ArticleModel

`ArticleModel` 是 Markdown 内容的稳定中间表示，不允许主题直接操作 Obsidian 生成的任意 DOM。

最低覆盖：

- 标题与层级。
- 普通段落和行内格式。
- 有序、无序和任务列表。
- 引用和 Callout。
- 代码块和行内代码。
- 表格。
- 图片、图注和链接。
- 脚注。
- 当前支持的自定义代码块组件。

### 5.2 LayoutPlan

`LayoutPlan` 由本地规则生成，用于描述内容和视觉组件的对应关系：

```ts
interface LayoutPlan {
  schemaVersion: 1;
  articleType: string;
  themeId: string;
  recipeId: string;
  sections: PlannedSection[];
  options: LayoutOptions;
}
```

首版不做自动内容理解。文章类型和配方由用户选择，规则只处理确定性的编号、目录、章节包装、结尾卡片和组件映射。

保留 `LayoutPlan` 的原因是隔离内容、排版决策和最终 HTML，而不是为 AI 预留接口。

### 5.3 ThemeManifest

新主题协议至少包含：

```ts
interface ThemeManifest {
  schemaVersion: 3;
  id: string;
  name: string;
  version: string;
  license: string;
  source?: ThemeSource;
  tokens: DesignTokens;
  components: ComponentMap;
  recipes: ArticleRecipe[];
  compatibility: CompatibilityProfile;
}
```

设计令牌至少包括：

- 主色、强调色、正文色、弱化色和背景色。
- 正文字号、行高和段落间距。
- 边框、圆角、阴影降级策略。
- 标题层级、代码、引用、列表、表格和图片规则。

所有增强组件必须从主题令牌读取颜色和间距，不再使用 `DEFAULT_ACCENT` 一类硬编码值。

### 5.4 组件与文章配方

首批标准组件：

- 文章引言。
- 目录。
- 章节标题。
- 步骤。
- 检查清单。
- 摘要卡。
- 引用卡。
- 作者卡。
- 关注引导。
- FAQ。
- 时间线。
- 对比表格。
- 代码块。
- 图片和图注。
- 文章结尾。

首批文章配方：

- 通用长文。
- 教程与步骤。
- 清单与方法论。
- 产品或工具介绍。
- 观点与评论。
- 周报与复盘。

配方只负责选择组件顺序和默认开关，不得改写用户正文。

### 5.5 微信兼容性验证

验证分为三层：

1. 主题导入验证：阻止非法主题进入主题库。
2. 渲染验证：在预览旁持续显示检查状态。
3. 复制验证：存在阻断错误时禁止直接复制。

严重级别：

- `error`：可能导致内容丢失、复制失败或严重错版，阻断复制。
- `warning`：可能存在平台差异，允许用户确认后继续。
- `info`：排版建议，不影响复制。

第一版检查项：

- 禁止 `script`、`style`、`iframe`、表单和事件属性。
- 输出片段不保留 `class`、`id` 和 `data-*`。
- 将非必要 `div` 改写为语义明确的 `section`、`p`、`span`、`figure` 等标签。
- 限制 `flex`、`grid`、`float`、固定定位、粘滞定位和横向滚动等脆弱布局。
- 使用内联 CSS，并按白名单过滤属性和值。
- 检查无效 URL、空图片、超大 Base64 图片和本地资源解析失败。
- 检查标题、段落、列表、链接和图片在转换前后的数量，防止静默丢失。
- 检查代码块换行、表格宽度、字号、行高和中文标点。
- 自动修复必须是保守操作，不得通过删除正文来消除错误。

### 5.6 复制与导出

继续支持：

- `text/html` 和 `text/plain` 剪贴板复制。
- 独立 HTML 片段导出。
- 本地长图导出。

新增：

- 1:1.33 等固定比例分段图导出。
- 复制前验证报告。
- 导出文件记录插件版本、主题版本和验证摘要。

图片策略：

- 不实现图片生成。
- 本地图片复制失败时给出明确错误，不静默跳过。
- 默认不把所有图片无条件转换为巨大 Base64。
- Base64 仅作为可配置的兼容策略，并设置单图和总载荷上限。

### 5.7 工作台界面

不照搬网页端三栏编辑器。Obsidian 当前文档已经承担编辑区。

窄侧栏使用三个标签：

- 预览。
- 主题与配方。
- 检查。

宽工作区使用双栏：

- 左侧：主题、配方、组件和历史。
- 右侧：微信公众号宽度预览及检查状态。

底部固定操作：

- 复制。
- 导出 HTML。
- 导出长图。
- 导出分段图。

### 5.8 手工主题工作室

主题工作室仅提供确定性的手工能力：

- 编辑设计令牌。
- 开关和调整标准组件。
- 选择文章配方。
- 使用固定示例文章实时预览。
- 导入和导出 `ThemeManifest`。
- 保存前执行主题验证。

不提供自然语言生成主题和参考图生成主题。

## 6. 主题治理与迁移

### 6.1 内置主题分层

不再以主题数量作为主要目标。

- `Core`：8–12 个经过完整验证的旗舰主题。
- `Classic`：从现有主题中迁移且通过验证的兼容主题。
- `Legacy`：只能使用 v2 适配器的主题，默认折叠展示。
- `Quarantine`：来源或许可证不清、验证失败的主题，不随正式版发布。

### 6.2 现有主题处理

对 94 个主题建立清单，逐项记录：

- 文件和主题 ID。
- 原作者和来源仓库。
- 原许可证。
- 转换脚本。
- 当前修改内容。
- v3 兼容性结果。
- 最终处置：Core、Classic、Legacy、重写或移除。

30 个标记为 `xiaohu` 的主题必须先确认上游许可证及归属。不能确认来源的主题不得进入 v3 正式发行包。

### 6.3 用户数据迁移

- 首次启动 v3 时备份完整 v2 设置。
- 为设置和主题数据增加 `schemaVersion`。
- 自定义 v2 主题优先转换为 v3 `LegacyThemeAdapter`。
- 迁移失败时保留原始数据并提示，不覆盖用户数据。
- v3 RC 之前保留“使用 v2 渲染器”的回退开关。
- 历史快照记录源文件哈希、主题版本、配方、插件版本和验证报告。

## 7. AGPL 迁移方案

### 7.1 许可证策略

- v2.0.11 作为最后一个 MIT 维护版本。
- `v3` 开发分支从第一个提交开始使用 `AGPL-3.0-or-later`。
- v3 的源代码、构建脚本、主题协议、内置主题和发布产物按 AGPL 发布。
- 如果复用 gzh-design-skill 或 WX-Typesetting 的 AGPL 代码、组件或主题，必须记录原文件、原作者、许可证、修改内容和对应提交。
- 架构思想可以参考，但不应复制其重复清洗器、巨型提示词和前端密钥方案等不适合本项目的实现。

### 7.2 原 MIT 代码的处理

当前代码来源于 MIT 项目并包含多名贡献者。迁移时：

- 保留原 MIT 许可证全文，建议存放为 `LICENSES/MIT-original.txt`。
- 主 `LICENSE` 改为完整 AGPL-3.0-or-later 文本。
- 新增 `NOTICE`，列出原项目、原作者、历任贡献者和主要上游来源。
- 新增 `THIRD_PARTY_NOTICES.md`，记录依赖、主题和素材许可证。
- 不删除现有版权声明。
- 对新增或重写文件使用统一 SPDX 头，例如：

```text
SPDX-License-Identifier: AGPL-3.0-or-later
```

MIT 许可允许在保留版权及许可声明的前提下再发布和组合，但许可证迁移不能解决来源不明素材的问题，因此主题和素材审计是正式发布的阻断条件。

### 7.3 发布合规

AGPL 发布工作流必须：

- 从干净检出的准确 Git tag 执行 `npm ci` 和生产构建。
- 自动核对 `package.json`、`manifest.json`、`versions.json` 和 Git tag。
- 上传 `main.js`、`manifest.json`、`styles.css`、`LICENSE`、`NOTICE` 和 `THIRD_PARTY_NOTICES.md`。
- 在 Release 中链接准确源码 tag 和构建说明。
- 保证用户能够获得与二进制产物对应的完整源代码。
- 对打包进入 `main.js` 的依赖生成许可证清单。
- 在 CI 中阻止缺少来源或许可证元数据的内置主题进入发行包。

许可证工作属于工程合规计划，不构成法律意见。正式发布前应由维护者再次确认 Obsidian API、打包依赖和主题素材的许可证兼容性。

## 8. 分阶段实施

### 阶段 0：v2.0.11 稳定性收口，4–6 个工作日

许可证：MIT，作为最后一个 MIT 版本。

交付：

- 增加文本与 JSON 解析检查，防止编码损坏进入发行包。
- 补回或重写版本同步脚本。
- 增加版本、manifest、tag 一致性检查。
- 建立测试框架和基础固定文章。
- 建立 94 个主题的来源与许可证台账。
- 为现有复制结果建立兼容性基线报告。
- 修正 Release 工作流，避免生成与源码版本不一致的 Release。

退出标准：

- TypeScript、构建和基础测试均通过。
- 版本脚本可在干净仓库中重复执行。
- 主题台账覆盖率 100%，允许暂时标记为“待确认”。
- 发布 v2.0.11，并标记为最后一个 MIT 版本。

### 阶段 1：v3.0.0-alpha，8–10 个工作日

许可证：AGPL-3.0-or-later。

交付：

- 完成 AGPL、NOTICE 和第三方声明迁移。
- 建立 `ArticleModel`、`LayoutPlan` 和 `ThemeManifest`。
- 拆分核心模块和 UI 模块。
- 实现 v2 设置、主题和渲染兼容层。
- 建立主题注册、版本和迁移机制。
- 重构增强组件，移除硬编码强调色。

退出标准：

- 标准 Markdown 内容可通过新管线渲染。
- 用户可切换新旧渲染器并得到完整内容。
- v2 设置备份和迁移可回滚。
- 所有新增源文件具备正确许可证标识。

### 阶段 2：v3.0.0-beta，7–9 个工作日

交付：

- 实现标签、属性、CSS 和内容完整性验证器。
- 实现复制阻断、警告确认和验证详情。
- 完成首批标准组件和 6 类文章配方。
- 完成 8–12 个 Core 主题。
- 建立主题 lint、固定样例和视觉快照测试。
- 改造图片处理，增加大小限制和错误提示。

退出标准：

- Core 主题对固定测试集不存在阻断错误。
- 转换前后标题、段落、列表、链接和图片数量一致。
- 验证器能够定位到具体节点、规则和修复建议。
- 复制失败不会产生“看似成功”的提示。

### 阶段 3：v3.0.0-rc，6–8 个工作日

交付：

- 完成响应式工作台。
- 完成手工主题工作室。
- 完成历史快照和恢复。
- 完成长图和分段图导出。
- 完成 Classic/Legacy 主题迁移。
- 清理或隔离来源不明、兼容性不合格的主题。
- 完成用户迁移说明和主题开发文档。

退出标准：

- 旧用户升级、回退和恢复流程通过测试。
- 手工主题导入、编辑、验证和导出形成闭环。
- 发布包中不存在来源未确认的主题和素材。
- v2 渲染器只作为显式兼容选项存在。

### 阶段 4：v3.0.0 正式版，4–5 个工作日

交付：

- Windows、macOS 和主流 Obsidian 版本回归。
- 微信公众号实际粘贴验收。
- 性能、内存和包体积优化。
- 完成 AGPL 发布合规检查。
- 发布 v3.0.0 和迁移指南。

退出标准：

- 所有 P0 缺陷关闭。
- CI 能从 tag 复现发行产物。
- Release 中包含许可证、NOTICE、第三方声明和源码链接。
- 正式版保留明确的回退路径。

## 9. 测试与验收标准

### 9.1 固定测试集

至少维护 30 篇固定文章，覆盖：

- 短文、长文和空文档。
- 多级标题。
- 中英文混排和标点。
- 各类列表和任务列表。
- 代码块、行内代码和长代码行。
- 表格、引用、Callout、脚注和链接。
- 本地图片、网络图片、无效图片和多图文章。
- 所有标准组件。
- 用户页头、页尾和自定义主题。

### 9.2 功能门槛

- Core 主题在固定测试集上的 `error` 数量为 0。
- 标题、段落、列表、链接和图片内容保留率为 100%。
- 阻断错误存在时无法误触直接复制。
- v2 设置和自定义主题迁移测试全部通过。
- 迁移失败不会破坏原始数据。
- 插件在完全离线环境中具备全部核心能力。

### 9.3 性能目标

- 1 万字普通中文文章的本地渲染目标小于 500ms。
- 输入连续变化时使用防抖和增量更新，避免重复完整渲染。
- 长图导出失败时释放 Canvas 和临时对象。
- 正式包体积不高于 v2.0.10 基线的 120%，超过时必须说明原因。

### 9.4 发布门禁

以下任一项失败都不得发布：

- TypeScript 检查。
- 单元测试和固定文章测试。
- 主题 schema、lint 和许可证检查。
- 内容完整性检查。
- 版本一致性检查。
- AGPL、NOTICE 和第三方许可证检查。
- 生产构建及发行文件校验。

## 10. 风险与应对

| 风险 | 影响 | 应对 |
|---|---|---|
| 主题来源或许可证不明确 | 无法合规发布 | 建台账；无法确认则移除或独立重写 |
| 微信编辑器行为变化 | 已验证样式失效 | 维护兼容性档案和真实粘贴回归集 |
| v2 自定义主题结构差异大 | 自动迁移失败 | Legacy 适配器、迁移备份和回退开关 |
| 过度清洗 HTML | 正文或格式丢失 | 内容计数、保守修复、禁止静默删除 |
| 组件重构引入视觉回归 | 用户体验退化 | 固定文章截图和主题快照比较 |
| 编码工具链或终端误判 | 误修改正常中文内容 | 使用 UTF-8/Unicode 解析检查，禁止无依据的全仓转码 |
| Release 与源码不一致 | 用户无法复现或回退 | tag 驱动构建、版本一致性门禁 |
| 大图和 Base64 导致内存问题 | 复制或导出失败 | 尺寸上限、分段处理和资源释放 |

## 11. 任务优先级

### P0：正式版阻断项

- 文本编码与 JSON 解析防护。
- 可复现版本和发布流程。
- AGPL 与第三方许可证合规。
- ArticleModel、ThemeManifest 和迁移层。
- 微信兼容性验证器。
- 内容完整性检查。
- Core 主题。
- 自动化测试与真实粘贴验收。

### P1：v3 正式版目标

- 文章配方。
- 标准语义组件。
- 响应式工作台。
- 历史快照。
- 手工主题工作室。
- 分段图导出。
- Classic 和 Legacy 主题整理。

### P2：v3.x 后续

- 更多经过审计的主题和文章配方。
- 主题包签名与社区主题导入。
- 更细致的差异比较和批量迁移工具。
- 微信兼容规则库的独立发布。

AI、模型接入和生成式图片不进入当前 v3.x 承诺范围；未来如需启动，应重新立项并单独完成隐私、成本、可靠性和产品价值评审。

## 12. 完成定义

只有同时满足以下条件，v3 升级才算完成：

1. 核心排版、预览、复制和导出完全离线可用。
2. 复制结果经过明确、可解释、可复现的兼容性验证。
3. 内容完整性达到验收标准。
4. v2 用户数据有可靠迁移和回退路径。
5. 主题数量不以牺牲来源、质量和兼容性为代价。
6. AGPL 源码、许可证、NOTICE、第三方声明和构建产物相互对应。
7. 正式 Release 能由准确 Git tag 在 CI 中重新构建。
8. 文档明确说明功能边界、数据处理方式、主题开发协议和迁移方法。
